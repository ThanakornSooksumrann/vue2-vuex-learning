Lifecycle ของ Vue Component

Component ทุกตัวมี "ช่วงชีวิต" ตั้งแต่เกิดจนตาย Vue จะเรียก method พิเศษ (lifecycle hook) ให้เราแทรกโค้ดเข้าไปในแต่ละช่วงได้ เรื่องนี้สำคัญมากเพราะโปรเจกต์นี้ใช้ mounted และ beforeMount เกือบทุกหน้าจอ เพื่อโหลดข้อมูลตั้งต้นและตั้งชื่อ title ของหน้า

1. ลำดับ Lifecycle Hook ทั้งหมด (Vue 2)

beforeCreate
ยังไม่มี data/methods ให้ใช้เลย (เกิดก่อน reactivity ถูกตั้งค่า)
↓
created
มี data / computed / methods ให้ใช้แล้ว แต่ ยังไม่มี DOM (this.$el ยังไม่มี) — เหมาะกับเรียก API เพื่อโหลดข้อมูลตั้งต้น
↓
beforeMount
เตรียม render ครั้งแรก DOM ยังไม่ถูกแปะเข้าหน้าเว็บจริง
↓
mounted
DOM พร้อมใช้งานแล้ว (this.$el / this.$refs ใช้ได้) — เหมาะกับสั่งงาน jQuery/Kendo widget, วัดขนาด element, โฟกัส input
↓ (วนซ้ำทุกครั้งที่ data เปลี่ยน)
beforeUpdate
data เปลี่ยนแล้ว แต่ DOM ยัง "เก่า" อยู่
updated
DOM re-render เสร็จแล้วตาม data ใหม่
↓ (เมื่อ component ถูกถอดออกจากหน้าจอ)
beforeDestroy
component ใกล้ตาย — จุดสำคัญสำหรับ cleanup: clearInterval, ปิด event listener, disconnect SignalR
destroyed
component ตายสมบูรณ์แล้ว ไม่มีอะไรให้ทำต่อ
Vue 3 ใช้ชื่อไม่เหมือนกัน (เผื่อไปเจอบทความ Vue 3 บนอินเทอร์เน็ต)
Vue 3 เปลี่ยนชื่อ beforeDestroy → beforeUnmount และ destroyed → unmounted แต่โปรเจกต์นี้เป็น Vue 2 ให้ใช้ชื่อ beforeDestroy / destroyed เท่านั้น

3. Hook ไหนควรใช้ทำอะไร (เทียบกับที่โปรเจกต์นี้ทำจริง)

Hookใช้ทำอะไรดีตัวอย่างในโปรเจกต์นี้
created เตรียมค่าตัวแปรที่ไม่ต้องพึ่ง DOM, เรียก action เพื่อดึงข้อมูล dropdown src/app/shared/Layout/MasterLayout.vue ใช้ created() โหลด MenuData เข้า sidebar
beforeMount เช็คเงื่อนไขก่อน render ครั้งแรก เช่นเช็ค URL ว่าเป็นหน้า login หรือไม่ src/App.vue เช็ค URL ว่าใช่ /SignIn หรือ /AutoSignIn หรือเปล่า ก่อนจะ mount
mounted โหลดข้อมูลเริ่มต้นของหน้าจอ, ตั้งชื่อ title, ผูก resize/interval, สั่งงาน widget แทบทุกหน้าจอ (view) เรียก this.$store.set("Layout_MasterLayout/setTitle", {...}) ใน mounted()
beforeDestroy เคลียร์ interval/timer, ปิด hub connection, ลบ event listener ที่ผูกไว้เอง App.vue มี logic ปิด SignalR hub connection ตอนเปลี่ยนหน้า (ใน router guard แทน)
ข้อผิดพลาดที่พบบ่อย
อย่าเรียก this.$refs.xxx ใน created() — ตอนนั้น DOM ยังไม่ถูกสร้าง $refs.xxx จะเป็น undefined เสมอ ต้องรอถึง mounted() เท่านั้นถึงจะใช้ $refs ได้

4. ตัวอย่างจริง: โหลดข้อมูล + polling + cleanup (pattern ที่เจอบ่อยที่สุดในงานจริง)

สถานการณ์ที่เจอแทบทุกวัน: หน้าจอ "ติดตามสถานะคำสั่งซื้อ/งานที่ส่งประมวลผล" ต้อง โหลดข้อมูลทันทีที่เข้าหน้า แล้ว poll (เรียกซ้ำเป็นช่วง ๆ) เพื่อเช็คสถานะล่าสุด จนกว่าผู้ใช้จะออกจากหน้านั้น — ถ้าลืมเคลียร์ setInterval ตอนออกจากหน้า จะเกิด memory leak และยังเรียก API ต่อไปเรื่อย ๆ ทั้งที่ผู้ใช้ไปหน้าอื่นแล้ว (บั๊กคลาสสิกที่เจอบ่อยมาก) 

export default {
  props: ["orderId"],
  data() {
    return {
      order: null,
      loading: true,
      error: "",
      pollTimer: null
    };
  },
  created() {
    // โหลดข้อมูลตั้งต้นทันทีที่ component ถูกสร้าง ไม่ต้องรอ DOM
    this.fetchOrder();
  },
  mounted() {
    // เริ่ม poll เช็คสถานะทุก 5 วินาที "หลังจาก" DOM พร้อมแล้ว
    this.pollTimer = setInterval(() => {
      this.fetchOrder({ silent: true }); // silent = ไม่โชว์ loading เต็มจอทุกรอบ
    }, 5000);
  },
  beforeDestroy() {
    // ต้องเคลียร์เสมอ ไม่งั้น interval จะยังทำงานต่อแม้ user ออกจากหน้านี้ไปแล้ว
    clearInterval(this.pollTimer);
  },
  methods: {
    async fetchOrder(opts = {}) {
      if (!opts.silent) this.loading = true;
      this.error = "";
      try {
        const res = await axios.get(`/api/orders/${this.orderId}`);
        this.order = res.data;
      } catch (err) {
        this.error = "โหลดข้อมูลไม่สำเร็จ กรุณาลองใหม่";
      } finally {
        this.loading = false;
      }
    }
  }
};

ด้านล่างคือ component เดียวกันแต่จำลอง API ด้วย setTimeout (แทน axios จริง) ให้กดดูการทำงานจริงได้ — สังเกตว่าพอกด "ทำลาย component" แล้ว log จะหยุดขึ้นทันที (เพราะ beforeDestroy เคลียร์ interval ไปแล้ว ไม่ใช่แค่ซ่อน UI เฉย ๆ)

สรุปสิ่งที่ต้องจำ

  • ลำดับ: beforeCreate → created → beforeMount → mounted → (beforeUpdate → updated)* → beforeDestroy → destroyed
  • เรียก API / โหลดข้อมูลตั้งต้น: ใช้ created หรือ mounted (ถ้าต้องใช้ $refs ต้อง mounted เท่านั้น)
  • ทำ polling (setInterval) ให้เริ่มใน mounted และเคลียร์ใน beforeDestroy เสมอ — คู่กันเป็นคู่ ห้ามลืมฝั่งใดฝั่งหนึ่ง
  • เคลียร์ของอื่น ๆ (event listener, hub connection) ก็ใช้ beforeDestroy เหมือนกัน ไม่งั้น memory leak
