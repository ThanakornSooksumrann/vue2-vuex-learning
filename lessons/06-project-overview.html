ส่วนที่ 2 · โปรเจกต์ KL.Core.UI 06/10

ภาพรวมสถาปัตยกรรมของ KL.Core.UI (Faco Core)

โปรเจกต์นี้ไม่ใช่ Vue app ธรรมดา ๆ ที่ generate จาก vue create — มันคือระบบ enterprise ขนาดใหญ่ที่มีการ "ประกอบร่าง" (compose) หลาย ๆ กลไกเข้าด้วยกัน เช่น auto-routing, auto-vuex-module, multi-instance embedding, webpack alias พิเศษ บทนี้จะไล่ตั้งแต่จุดเริ่มโหลดแอปจนถึงกลไกพิเศษที่ควรรู้ก่อนเริ่มเขียนโค้ด

1. Stack เต็ม ๆ ของโปรเจกต์นี้

2. ลำดับการบูตแอป (Boot sequence)

src/main.js
Entry point จริง — import polyfill, เตรียม adapter component (b-row/b-col/b-card) แล้วเรียก CreateVueInstance
↓
src/CreateVueInstance.js
"ห้องเครื่อง" ของแอป — import + register Kendo ทุกตัว, Bootstrap-Vue, Vuelidate, ตั้ง global mixin, ผูก router+store+i18n แล้ว new Vue({...})
↓
src/App.vue
Root component จริง — เช็คว่าเป็นหน้า login/public/authorized หรือเปล่า แล้วค่อยแสดง MasterLayout
↓
MasterLayout.vue
Shell หลัก: header + เมนู sidebar (kendo-treeview) + <router-view> ที่แสดงหน้าจอปัจจุบัน (ดูบทที่ 08)
// src/main.js (ของจริง — ย่อบางส่วน)
import 'babel-polyfill';
import * as cmpCore from "faco.core@shared/Adapter/Bootstrap/index.js";

const adapterCmp = {
  components: {
    'b-col': cmpCore.BCol, 'b-row': cmpCore.BRow,
    'b-card': cmpCore.BCard, 'b-container': cmpCore.BContainer,
    'b-app': cmpCore.BApp
  }
};

import initialVMInstance from "./CreateVueInstance";
initialVMInstance(null, adapterCmp).$mount("#app-root");
สังเกต: initialVMInstance(null, adapterCmp) — พารามิเตอร์ตัวแรกทำไมเป็น null?
เพราะ CreateVueInstance.js ถูกออกแบบมาให้เรียกได้ 2 แบบ: (1) เป็นแอปหลักตัวเดียว (parameter แรก = null ตามปกติ) หรือ (2) เป็น "แอปย่อยที่ฝังอยู่ในแอปหลักอีกที" (ส่ง vmMaster ของแอปหลักเข้าไป) — อธิบายต่อในหัวข้อถัดไป

3. Global mixin และ Prototype property — เครื่องมือที่ใช้ได้ทุก component ทันที

CreateVueInstance.js ผูกของพวกนี้ไว้กับทุก component โดยอัตโนมัติ ไม่ต้อง import เอง:

ชื่อคืออะไรใช้ยังไง
this.$utils Utility รวมของระบบ (singleton) เช่น อ่าน/เขียน localStorage, แจ้งเตือน, validate, hash this.$utils.addNotify("บันทึกสำเร็จ", 200), this.$utils.getStorage("token")
this.$store Vuex store (มาตรฐาน Vue) — ใช้คู่กับ pathify get/set ดูบทที่ 07
Global mixin methods (_getMainApp, _onSuccessUploadFile, ฯลฯ) ฟังก์ชันช่วยเรื่อง upload ไฟล์, จัดการ SignalR hub, validate tag ห้ามใส่ในช่อง input เรียกใช้ได้ตรง ๆ ทุก component เช่น @success="_onSuccessUploadFile" บน kendo-upload
$configCore ตัวแปร global (ไม่ใช่ของ Vue) เก็บ config ต่อ environment เช่น URL ของแต่ละ API, ชื่อแอป, isDebug $configCore.hostApi.financeApi, $configCore.appName

4. Multi-instance pattern (vmMaster) — จุดที่ไม่เหมือนโปรเจกต์ Vue ทั่วไป

บางกรณีระบบนี้ต้องฝัง Vue instance หนึ่งไว้ในอีก instance หนึ่ง (เช่น เปิดโมดูลย่อยเป็น embedded widget ภายในแอปหลัก) ฟังก์ชัน CreateVueInstance เลยรับ vmMaster เป็น parameter — ถ้ามีค่า แปลว่า instance นี้เป็น "แอปลูก" ที่ถูกฝังอยู่ในแอปแม่ 

// src/CreateVueInstance.js (โค้ดจริง)
export default (vmMaster, adapterCmp) => {
  // ถ้าเป็น external (แอปลูกที่ฝังอยู่ในแอปแม่) ให้เอา route ของตัวเองไปรวมกับ router ของแอปแม่
  if ($configCore.__isExternal) {
    const routes = router.options.routes || [];
    vmMaster.$router.options.routes.push(...routes);
    vmMaster.$router.addRoutes(routes);
  }

  const useRouter = ($configCore.__isExternal ? vmMaster.$router : router);

  const vmInstance = new Vue({
    router: useRouter,
    i18n,
    store,
    render: h => h(App)
  });
  vmInstance.vmMaster = vmMaster;   // เก็บ reference กลับไปหาแอปแม่ไว้ใช้ทีหลัง
  return vmInstance;
};
สิ่งที่ต้องรู้ในทางปฏิบัติ
ในโค้ดของหลาย component จะเห็น pattern แบบ const vmMaster = this.$root.vmMaster; if (vmMaster) { ... } else { ... } อยู่บ่อย ๆ — นั่นคือการเช็คว่า "ตอนนี้ component รันอยู่ในโหมดแอปเดี่ยว หรือถูกฝังอยู่ในแอปแม่" แล้วเลือกพฤติกรรมให้เหมาะสม (เช่น จะ dispatch action ไปที่ store ของตัวเอง หรือส่งต่อให้แอปแม่จัดการแทน) ถ้าเจอโค้ดแบบนี้ไม่ต้องแปลกใจ มันจงใจทำแบบนี้

5. Webpack alias พิเศษ: faco.core@...

ทั้งโปรเจกต์ import ไฟล์กันด้วย alias รูปแบบ faco.core@app/..., faco.core@shared/..., faco.core@views/... แทนการเขียน relative path (../../../shared/xxx) ที่ยาวและงงเวลาย้ายไฟล์ — alias พวกนี้ถูกตั้งไว้ใน webpack config แล้ว ให้จำ mapping คร่าว ๆ ไว้ดังนี้:

Aliasชี้ไปที่โฟลเดอร์จริง
faco.core@sharedsrc/app/shared
faco.core@viewssrc/app/views
faco.core@appsrc/app
faco.core@repositorysrc/app/repositories/repository.js
faco.core@assetssrc/app/assets
faco.core@srcsrc

6. Build แบบแบ่ง module (SCOPE_MODULE)

โปรเจกต์นี้ใหญ่มาก (มีทั้ง Finance, Collection, Legal, HPAdmin, PostDateCheque ฯลฯ) จึงมี script npm หลายตัวใน package.json เพื่อรัน dev server โดยจำกัดขอบเขต module ที่ build (ลดเวลา build/reload ตอน dev): 

npm run local              // รันทุก module (เต็มระบบ)
npm run local-finance      // รันเฉพาะ Finance
npm run local-hpadmin      // รันเฉพาะ HPAdmin
npm run local-main         // รันเฉพาะ Finance,HPAdmin,Legal,Collection
คำแนะนำ
ถ้ากำลังพัฒนาหน้าจอใน module ไหน ให้รัน script ที่ตรงกับ module นั้น (หรือ local-main ถ้าทำหลาย module) จะทำให้ webpack-dev-server build เร็วขึ้นมากเทียบกับ npm run local ที่ build ทั้งระบบ

สรุปสิ่งที่ต้องจำ

  • ลำดับบูต: main.js → CreateVueInstance.js → App.vue → MasterLayout.vue → router-view
  • this.$utils, global mixin methods, และ $configCore ใช้ได้ทุก component ทันทีโดยไม่ต้อง import
  • รู้จัก pattern vmMaster ไว้ — เจอโค้ดเช็ค this.$root.vmMaster อย่าตกใจ
  • import ไฟล์ด้วย alias faco.core@shared, faco.core@views, faco.core@app แทน relative path ยาว ๆ
  • เลือกใช้ npm script ให้ตรงกับ module ที่กำลังทำ เพื่อความเร็วตอน dev
